# 边-云协同IoT智能环境监测系统 — 实验报告

## 一、系统架构

本系统采用端-边-云三层协同架构：

```
┌─────────────────┐      HTTP POST       ┌─────────────────┐      HTTP POST       ┌─────────────────┐
│   VM1: 终端层    │ ──────────────────→  │   VM2: 边缘层    │ ──────────────────→  │   VM3: 云端层    │
│                  │                      │                  │                      │                  │
│ • 多设备模拟      │    JSON 传感器数据    │ • 异常检测       │   异常/聚合数据       │ • 数据存储       │
│ • 定时采集发送    │                      │ • 定时聚合       │                      │ • REST API      │
│ • 多线程并发      │                      │ • 数据转发       │                      │ • Web 仪表盘    │
└─────────────────┘                      └─────────────────┘                      └─────────────────┘
  Python + Requests                         Flask + Requests                         Flask + SQLite + Chart.js
```

## 二、各模块设计与实现

### 2.1 VM1 终端模拟器 (sensor_simulator.py)

**功能**：模拟多个 IoT 传感器设备，定时生成环境监测数据并发送至边缘节点。

**实现要点**：
- 使用多线程实现多设备并发模拟（默认3个设备：sensor_01、sensor_02、sensor_03）
- 每5秒生成一组随机数据，范围设定合理：温度15-40°C、湿度30-90%、PM2.5 10-80μg/m³、CO2 300-600ppm
- 数据格式为 JSON，包含 device_id、timestamp 及四项环境指标
- 通过 HTTP POST 发送至 VM2 的 /api/data 接口
- 发送失败时打印错误日志并继续运行，保证系统鲁棒性

### 2.2 VM2 边缘预处理 (edge_server.py)

**功能**：接收终端数据，执行异常检测与数据聚合，选择性转发至云平台。

**实现要点**：
- Flask Web 服务，监听 POST /api/data 接口
- **异常检测规则**：PM2.5 > 50μg/m³ 或 温度 > 38°C 时判定为异常，标记 anomaly=true 并附 anomaly_reason
- 异常数据**立即转发**至云端，不等待聚合窗口
- 正常数据存入内存缓冲池，每30秒聚合一次：按设备计算各项平均值得出
- 聚合数据标记 aggregated=true 和 aggregate_count，发送至 VM3
- 使用线程锁保证多线程下缓冲池的数据安全

### 2.3 VM3 云端平台 (cloud_server.py)

**功能**：接收边缘数据，持久化存储，提供 Web API 和可视化仪表盘。

**实现要点**：
- Flask Web 服务，提供以下接口：
  - `POST /api/data` — 接收并存储数据到 SQLite
  - `GET /api/data/latest?limit=N&device=X` — 获取最新数据
  - `GET /api/devices` — 获取所有设备列表
  - `GET /api/stats` — 获取统计摘要（设备数、数据总量、异常数、各设备最新值）
  - `GET /` — Web 仪表盘页面
- SQLite 数据库（sensor_data 表），包含设备信息、传感器数据、异常标记、聚合标记等字段
- 使用 WAL 模式提升并发读写性能

### 2.4 Web 仪表盘 (dashboard.html + dashboard.js)

**功能**：实时可视化展示监测数据。

**实现要点**：
- 采用**深色玻璃拟态 (Glassmorphism)** 设计风格，动态渐变光晕背景 + 网格纹理，视觉现代炫酷
- 使用 Chart.js 渲染实时折线图，支持温度/湿度/PM2.5/CO₂ 四个指标切换
- **每个设备分配固定的高辨识度颜色**，多设备折线一目了然；异常数据点以红色高亮突出显示
- 设备列表面板：展示各设备最新读数及状态（正常/异常），带设备色标，点击可筛选图表数据
- 统计卡片（带数字滚动动画）：在线设备数、数据总量、异常记录数、**边缘聚合上报次数**
- 异常事件列表：展示最近20条异常记录及其原因
- 每3秒自动轮询刷新数据，连接中断时状态指示灯变红提示

## 三、部署说明

### 3.1 Docker Compose 部署（推荐）

```bash
cd cloud-edge-iot
docker compose up -d
# 访问 http://localhost:5000 查看仪表盘
```

三个容器通过自定义 bridge 网络（iot-network）互联，服务名即为容器名（vm1-terminal、vm2-edge、vm3-cloud）。

### 3.2 分机部署

三台 Ubuntu 虚拟机需在同一内网，按 VM3 → VM2 → VM1 顺序启动：

| 虚拟机 | 启动命令 | 需配置的环境变量 |
|--------|----------|-----------------|
| VM3 | `cd vm3-cloud && python cloud_server.py` | CLOUD_HOST, CLOUD_PORT |
| VM2 | `cd vm2-edge && python edge_server.py` | CLOUD_URL=http://VM3_IP:5000 |
| VM1 | `cd vm1-terminal && python sensor_simulator.py` | EDGE_URL=http://VM2_IP:5000 |

## 四、功能验证

### 4.1 正常数据流验证
1. 启动 VM3 云平台，确认监听端口 5000
2. 启动 VM2 边缘节点，确认打印"边缘节点启动"及配置参数
3. 启动 VM1 终端模拟器，观察日志输出三个设备的发送记录
4. VM2 日志应显示"接收 正常"并每30秒输出"聚合转发"
5. VM3 日志应显示"存储 聚合"
6. 浏览器打开 VM3 仪表盘，确认实时图表更新

### 4.2 异常检测验证
1. 将 VM2 的 PM25_THRESHOLD 临时调低（如设为 20）
2. 重启 VM2，使大部分数据触发异常
3. 观察 VM2 日志出现"接收 异常"和"异常转发"
4. VM3 仪表盘上图表出现红色异常标记点
5. 异常事件列表中出现对应记录

### 4.3 多终端验证
1. 将 VM1 的 DEVICE_COUNT 设为 5
2. 重启 VM1，确认日志出现 sensor_01 ~ sensor_05 五个设备
3. 仪表盘设备列表显示5个设备
4. 点击各设备卡片可筛选对应数据

## 五、加分项实现

| 加分项 | 实现情况 |
|--------|---------|
| Docker 化部署 | 三个模块均编写 Dockerfile，提供 docker-compose.yml 一键编排 |
| 多终端模拟 | VM1 支持可配置数量（DEVICE_COUNT）的多线程设备模拟 |
| Web 展示 | VM3 提供基于 Chart.js 的实时数据仪表盘，含图表、设备管理、异常列表 |

## 六、总结与收获

通过本项目，实践了以下技能：
1. **多节点系统部署与联调**：三台虚拟机/容器之间的网络通信配置与联调
2. **边缘计算架构设计**：理解了边缘节点在数据预处理、异常检测、聚合转发中的角色
3. **IoT 数据处理与可视化**：从数据生成、传输、存储到 Web 实时展示的完整链路
4. **容器化部署**：Docker + Docker Compose 实现环境标准化和一键部署
5. **异常检测逻辑**：基于阈值的实时异常判断与优先转发机制

系统中仍可改进的方向包括：使用 MQTT 协议替代 HTTP 以获得更低延迟；引入时序数据库（如 InfluxDB）处理大规模数据；使用 WebSocket 替代轮询实现真正的实时推送。
